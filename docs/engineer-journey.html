01 Flow Overview

The engineer experience is organized into three distinct layers. Each layer answers one question and leads to the next.

Layer 1
Dashboard
“What should I work on?”
  • View assigned projects as a prioritized queue
  • See status, config progress, and last activity per project
  • Search / filter by status, client, or project name
  • Recently opened projects as quick-access cards
  • Access parts catalogue for reference and project planning
Layer 2
Project Hub
“What’s the context for this job?”
  • 5-second orientation: title, client, location, team, status
  • Status strip with 5 lifecycle stages
  • Config progress and quick stats (area, volume, cost)
  • Preview cards for Design, BOM, Engineering, Estimate
  • Deep-link CTAs into specific workspace tabs
Layer 3
Workspace
“Design, engineer, and deliver.”
  • Full CAD-style interface (canvas, inspector, toolstrip)
  • New/empty workspace shows templates gallery + blank project option
  • 7 workspace tabs: Design, Summary, Engineering, Estimate, BOM, Deliverables, History
  • Breadcrumb: ← Dashboard / Project Name
  • Autosave with version history in the History tab

Save Behavior

  • Autosave: work is automatically saved at regular intervals. Status bar shows “Saved” indicator with timestamp.
  • Manual save points: engineer can create named save points (e.g. “Before plumbing reroute”) from the History tab.
  • Version restore: any previous version can be restored from the History tab. Current state is preserved as a version before restore.
  • Browser crash recovery: autosave ensures work is recoverable. On next login, a toast notification shows: “Your last session was recovered.”

Configuration Drawer

A guided flow of 13 configuration steps across 6 groups. The drawer slides in from the left, overlaying the canvas.

  • Entry: engineer clicks “Configure” in the mode switcher or from the Summary tab
  • Navigation: step-by-step with back/next buttons. Steps can be completed in any order by clicking the step list.
  • Validation: each step shows validation status (complete, incomplete, error). Incomplete steps are highlighted.
  • Exit: close the drawer to return to the canvas. Progress is saved automatically.
  • Sync: config progress shown in the drawer matches the “8 / 13” indicator in the status bar and on the Dashboard.

What Changes from Current State

  • Breadcrumb in top bar: ← Dashboard / Project Name — clear escape hatch without browser controls
  • Remove userMode toggle: role is determined at login, not switched mid-session inside the workspace
  • Templates on empty workspace: when a workspace is new or empty, show a template gallery with pre-built pool designs and a “Start Blank” option
  • Catalogue access: engineer can browse the master parts catalogue from within the workspace (via the Components panel)
  • Autosave indicator: status bar shows save state so the engineer always knows their work is safe
  • Everything else stays as-is — canvas, panels, config drawer, workspace tabs, status bar

06 Key Flows

Core journeys the engineer performs inside the proposed 3-layer experience. Each flow includes error handling.

Flow A — Pick Up an Assigned Project

  1. Engineer lands on the Dashboard and scans assigned projects by status, progress, and recent activity.
  2. Engineer uses search or filters to narrow the queue if needed.
  3. Engineer opens a project and lands in the Project Hub for quick orientation.
  4. Engineer reviews context: client, location, team, lifecycle state, and quick stats.
  5. Engineer opens the full Workspace or deep-links straight into Engineering, BOM, or Estimate.

Error Handling

  • Project data is stale or loading fails: show a retry banner: “Unable to load project data. Retry?” with a retry button.
  • Project was unassigned while engineer was away: show a message: “This project is no longer assigned to you.” Return to Dashboard.

Flow B — Start Work on an Empty Project

  1. Engineer opens an assigned project from the Dashboard.
  2. Project Hub confirms the project is still early-stage (status: Draft, config progress: 0/13).
  3. Engineer opens the Workspace and sees a template gallery or a “Start Blank” option instead of an empty canvas.
  4. Engineer selects a template or starts blank. The workspace initializes with the chosen starting point.
  5. Engineer begins geometry, configuration, and component planning.
  6. Engineer uses Summary and Engineering tabs to validate progress before moving to the next stage.

Error Handling

  • Template gallery fails to load: fall back to “Start Blank” with a message: “Templates are temporarily unavailable. You can start with a blank project.”
  • Template application fails: show error toast and remain in the gallery. Engineer can retry or start blank.

Flow C — Produce Outputs and Handoff

  1. Engineer completes the design and reviews engineering checks in the Engineering tab.
  2. Engineer opens Estimate and BOM to validate costs and part selections.
  3. Engineer opens Deliverables to generate plans, specs, and exportable documentation.
  4. Engineer clicks “Mark as Ready for Estimate” in the status strip or from the Deliverables tab. This is a manual, explicit action — not automatic.
  5. A confirmation dialog appears: “Move this project to Ready for Estimate? This will notify the admin.”
  6. Project status changes to Ready for Estimate. The admin receives a notification.

Error Handling

  • Deliverables generation fails: show error toast with details: “Unable to generate [document name]. Check engineering compliance.” Link to the Engineering tab.
  • Status change fails: show error toast: “Unable to update project status. Please try again.” Project remains in current status.
  • Incomplete configuration blocks handoff: if config progress is not 13/13, show a warning: “Configuration is incomplete (8/13). Complete all steps before marking as ready.” Link to the Summary tab.

09 Edge Cases & Error States

Documented edge cases and expected behavior for each.

Scenario Expected Behavior
First-time engineer — no projects assigned Dashboard shows empty state: “No projects yet. Contact your admin or browse the catalogue.”
Engineer has 100+ assigned projects Dashboard supports pagination or infinite scroll. Search and filter are essential at this scale.
Engineer opens a project that was just unassigned by admin Show message: “This project is no longer assigned to you.” Return to Dashboard.
Browser crashes or tab closes during design work Autosave recovers work. On next login, toast: “Your last session was recovered.”
Engineer tries to mark project as Ready for Estimate but config is incomplete Block the status change. Show warning: “Configuration is incomplete (8/13). Complete all steps first.” Link to Summary.
Deliverables generation fails Show error toast with details and a link to the Engineering tab to resolve compliance issues.
Template gallery fails to load on empty workspace Fall back to “Start Blank” with a message explaining templates are temporarily unavailable.
Two engineers are assigned to the same project Both see the project on their Dashboard. Simultaneous editing behavior is TBD (see Open Decisions).
Admin and engineer work on the same project simultaneously Same as multi-engineer case — TBD. At minimum, last-write-wins with autosave. Presence indicators recommended.
Config progress shows 0/13 on an old project (legacy data) Show config progress as “Not configured” instead of “0 / 13”. Prompt engineer to run the configurator.
Client name is clicked but client data is unavailable Show a minimal card: “Client information is unavailable.” No navigation to admin views.

10 Open Decisions

Product questions that need team alignment before design can proceed.

Decision Options Recommendation
Should the lifecycle include On Hold and Cancelled statuses? A) Keep 5 stages (current)
B) Add On Hold and Cancelled as branch states		 Add them as branch states. On Hold pauses the project; Cancelled is a terminal state. Both should be admin-only actions.
How does multi-user collaboration work? A) Last-write-wins (simple, no locking)
B) Presence indicators + soft locks
C) Real-time co-editing (complex)		 Start with (A) last-write-wins + autosave. Add (B) presence indicators as a fast follow. (C) is a future investment.
Can the engineer create projects? A) No — admin only (current assumption)
B) Yes — engineer can create and self-assign		 Keep as admin-only for now. The empty state already says “Contact your admin.”
How does login route users to the right role? A) Single URL, role detected server-side
B) Separate URLs per role
C) Role selector after login		 Single URL with server-side role routing. Consistent with admin journey recommendation.
What notifications does the engineer receive? A) In-app only
B) Email only
C) Both, configurable per event type		 (C) Both, configurable. Key events: new assignment, status change by admin, project comment/mention.
Should the engineer see a read-only version of company billing? A) No — billing is admin-only
B) Read-only project cost on Project Hub		 (B) Read-only project cost is already on the Hub via the Estimate preview card. No separate billing access needed.
