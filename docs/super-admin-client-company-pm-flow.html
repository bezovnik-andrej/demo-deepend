Super Admin Flow: Client Company -> Project Manager

Derived from the current Super Admin and Engineer journey docs, with explicit assumptions where the product model is still undefined.

v0.2 · Updated per client feedback

01. Goal

Describe the end-to-end path for a Super Admin to onboard a client company, create or invite its primary project manager, and create the first project that connects them.

Success state: A client company exists in the Back Office, a project manager is associated with that company, a project has been created, and the project manager is visible as a collaborator or owner at the project level.

02. Flow Overview

This keeps the existing three-layer admin model intact: Back Office list views, detail views, then project-level access.

Entry
Admin Back Office
  • Super Admin logs into the Back Office.
  • Navigates to Companies to start onboarding a client company.
↓
Decision
New client or existing company?
  • If new: start the existing Add Company wizard.
  • If existing: open Company Detail and skip to project manager setup.
↓
Company Setup
Add Company Wizard
  • Step 1: Company Info
  • Step 2: Contract Signer Info
  • Step 3: Signing a Contract
↓
Client Contact Setup
Create or invite Project Manager
  • Super Admin adds the primary client-side operating contact.
  • This person becomes the default project manager for the company or for the first project.
↓
Project Setup
Create Project
  • Super Admin creates the project from the Projects section.
  • 3-step wizard: Select Company → Project Info → Confirm.
  • No team assignment during creation. PM added after via Collaborators tab.
↓
Exit
Project Detail
  • Overview confirms company and project basics.
  • Collaborators confirms the assigned project manager.
  • Admin can then open the full workspace if needed.

04. Step-by-Step Flow

This is the recommended task-level sequence using only the product structures already implied in the current docs.

1Open Back Office

The Super Admin lands in the Back Office and chooses Companies from the primary navigation.

2Check whether the client company already exists

If the company is already present, open its detail view. If not, start the Add Company flow that already exists in the admin journey.

3Create the client company

Use the existing three-step wizard: Company Info -> Contract Signer Info -> Signing a Contract. This establishes the company record and commercial relationship.

4Add the primary client-side project manager

From the company context, create or invite the person who will act as the day-to-day owner on the client side. Since a dedicated Project Manager section does not exist in the current IA, this should live under Company Detail or as part of a company contact list.

5Create the project

Go to Projects, start the Create Project flow, and attach the new project to the client company.

6Assign the project manager to the project

Use the project-level collaborator model already implied by the Collaborators tab in Project Detail. The project manager should appear there as the primary client-side collaborator.

7Verify in Project Detail

The project should now show the client company in the overview and the project manager in collaborators before the admin moves into the workspace.

05. Decision Points

Decision Options Expected Outcome
Does the client company already exist? Create a new company / use the existing company Avoid duplicate company records and keep projects tied to one company object.
Is there already a client-side project manager? Create or invite a new PM / select an existing company contact Every project gets a clear client-side owner.
When should the PM be assigned? During company setup / during project creation Prefer assignment during project creation so the same company can support multiple PMs across projects.
Where does the PM live in IA? Company Detail contacts / project collaborators only Best current fit is company-level contact first, then project-level collaborator assignment.

06. Edge Cases

Scenario Expected Behavior
Company already exists but contract is incomplete Allow the admin to open Company Detail, resume commercial setup, then continue to project creation.
Project manager is unknown at company onboarding time Allow company creation without PM assignment, but block project handoff until a PM is chosen.
One company has multiple project managers Store them at company level and require one primary PM per project during project setup.
PM changes after project creation Update the project from the Collaborators tab without recreating the company or project.
Admin creates project before company exists Redirect into company creation first or provide an inline "Add Company" shortcut from the project flow.

07. Assumptions and Gaps

These are the places where the current docs do not define the behavior yet, so this flow makes the minimum viable assumptions.

  • Project Manager is not yet a defined role in the current journey docs, so this flow treats the PM as a client-side collaborator tied to a company and then a project.
  • Company Detail now has 3 tabs (Overview, Contacts, Projects). The Contacts tab is the logical place for PM management.
  • Create Project is now a 3-step wizard (Select Company → Project Info → Confirm). PM assignment happens after creation via the Collaborators tab, not during the wizard.
  • Invite Company exists as an alternate path, but its internals are undefined. This document assumes it can eventually land on the same company-and-PM setup state.

08. Recommendation

Recommended product model: keep the current admin IA (2 sections: Companies, Projects), use the Contacts tab in Company Detail for PM management, and assign the PM from the Collaborators tab in Project Detail after project creation. This gives a clean hierarchy: Company → Contacts → Project → Collaborators.
